# Review-gate rejection calibration plan — 2026-06-14

## Context

Human review cleared 30 pending/open Idea Maze research runs. Kostya rejected all 30. This is calibration evidence: the review queue was dominated by polished but weak opportunity drafts.

Observed diagnostics after review:

- Pending runs dropped to 0.
- 30/30 reviewed runs were marked `validation_status: valid` and generated by `claude-sonnet-4-6`.
- 21/30 had current `final_score >= 9` even after taste recomputation; several were 10/10 before review.
- Source item count was thin: 20/30 had only 3–4 source items; 26/30 had <=5.
- Many repeated refs occurred across runs, especially generic Tavily search results around "self-serve" and repeated Reddit sources across duplicate clusters.
- Common human reject reasons: source contamination, thesis/source mismatch, duplicate clusters, one-good-source inflation, self-promo/founder-validation posts, weak willingness-to-pay, crowded generic wrappers, and consumer/education/travel low-WTP niches.

## Diagnosis

This is primarily a pipeline calibration issue, with source quality as an input problem.

The source pool contains useful signals, but the current pipeline can:

1. Promote weak or promotional Reddit posts as if they were buyer pain.
2. Mix unrelated sources into a coherent-sounding draft.
3. Reuse generic Tavily search refs across many opportunities.
4. Score high from semantic coherence / broad market language rather than direct pain, budget, and buyer specificity.
5. Let duplicates of rejected clusters re-enter review.
6. Treat `validation_status: valid` as prose/schema validity, not opportunity quality.

## Proposed improvements

### 1. Separate evidence card from opportunity draft

Before LLM synthesis, build a structured evidence card per candidate:

- buyer/user
- pain quote(s)
- current workaround
- time/money loss
- WTP/budget/tool-shopping evidence
- incumbent alternatives
- independent source count
- source contamination flags
- self-promo/founder-validation flags

Only generate a full research draft if the evidence card passes thresholds.

### 2. Evidence quality gate before research

Add/extend deterministic gating in `src/idea-maze/scripts/lib/evidence-quality.ts` and integrate it in `process-opportunities.ts` before research.

Suggested policy:

- Require at least 3 independent source refs for review-gate, or at least 2 with direct WTP/budget/time-loss.
- Cap score to 7 if independent source count < 3.
- Cap score to 7 if no direct WTP, budget, tool-shopping, time-loss, or current workaround signal.
- Cap score to 6 if promo/self-validation sources dominate.
- Cap score to 6 if the idea is a generic AI wrapper or generic founder meta-product.
- Quarantine source-contaminated drafts instead of review-gate.

### 3. Search-result hygiene

Tavily/search refs should not count as independent buyer evidence unless they are directly relevant and non-generic.

Rules:

- Penalize repeated generic search refs across opportunities.
- Ignore broad keyword matches like generic "self-serve" pages.
- Use search refs as context/incumbent research, not primary pain evidence.
- Require Reddit/inbox/direct forum pain sources for the pain side of evidence.

### 4. Duplicate/rejected-cluster gate

Before creating a review-gate run:

- Compare title/slug/thesis/source refs against rejected opportunities.
- If high shared refs or high semantic/token similarity, archive as `duplicate_rejected_opportunity`.
- Attach new evidence to existing clusters only if it materially improves quality; otherwise skip.

### 5. First-class reject reasons

Human-review state should store labels, not just `ignored`.

Initial labels:

- `source_contamination`
- `duplicate`
- `weak_wtp`
- `self_promo_or_founder_validation`
- `crowded_no_wedge`
- `consumer_retention_risk`
- `thesis_source_mismatch`
- `generic_ai_wrapper`
- `not_founder_fit`

Use these in taste/scoring and in diagnostics.

### 6. Better status diagnostics

Add a status/inspection command that reports:

- review-gate candidates by quality disposition
- count of candidates capped/archived by evidence gate
- top repeated source refs
- fallback/template/mismatch flags
- average score of recently rejected human-review runs
- most common rejection labels

## Suggested implementation sequence

1. Add diagnostics command for current review-gate quality and repeated refs.
2. Add reject-reason labels to human-review application flow.
3. Add deterministic duplicate/rejected-cluster gate.
4. Tighten evidence-quality gate thresholds and score caps.
5. Change research validation to distinguish prose-valid from opportunity-valid.
6. Re-run pipeline and review a much smaller queue; target <=5 review candidates/day with higher precision.

## Source-expansion ideas from external 30-point checklist

The current engine already covers part of the checklist:

- Reddit niche/friction mining via configured subreddits.
- Gmail and Telegram ingestion, though these are tiny compared with Reddit/search.
- Tavily web-search enrichment during research.
- Existing quality/duplicate gates in `process-opportunities.ts`, but thresholds and source hygiene are still too permissive.

Useful additions for this engine, prioritized:

1. GitHub issues / OSS discussions for recurring developer pain.
2. Chrome Web Store 1-star review mining for extension/workflow complaints.
3. SaaS feature-request / public roadmap mining for delayed features.
4. API/changelog monitoring for new integration windows.
5. Zapier/Make/n8n recipe mining for recurring automatable workflows.
6. Upwork vertical task mining for high-value repetitive services.
7. YouTube tutorial-comment mining for unresolved setup/workflow blockers.
8. Job-listing mining for repeated tool requirements and skill-friction patterns.
9. Public Notion/template directory mining for spreadsheet/template workflows that can become vertical agents.
10. LinkedIn/manual-data-task mining if access/rate limits are acceptable.

Less suitable as first moves:

- Generic "AI for X" search-query mining: likely noisy unless paired with buyer/WTP checks.
- Product Hunt reverse engineering: useful for inspiration but weak pain evidence.
- Broad podcast transcript mining: high processing cost; useful only for targeted verticals.
- Niche directory rebuilds as AI concierge services: often GTM-heavy and not naturally self-serve SaaS.

## Success criteria

- Review queue precision improves: at least 1–2 serious candidates per 10 reviewed.
- Duplicate rejected themes do not reappear as fresh review-gate runs.
- Generic search refs no longer inflate source independence.
- High final score implies direct pain + buyer + WTP/current workaround, not just coherent prose.
- Pending review queue remains small and actionable.
